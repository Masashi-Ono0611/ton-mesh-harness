What is this · TL;DR

The frontend can’t be taken down if there’s no server to take down.

A CLI that ships any static build to TON Storage and a .ton domain — in a single command, with zero infra setup.

DeFi UIs keep getting silenced — Tornado Cash, Uniswap, 1inch, Balancer. The contracts live on; the websites die because they sit on regular servers and regular DNS. TON already solved this with Storage, DNS, ADNL Tunnel, and the Payment Network — the same digital-resistance stack Resistance Tools / foundation.ton itself relies on. Sovereign Deploy Kit is the npx-style CLI for that stack — built for engineers, not GUI users.

~/projects/your-dapp · zsh 
$ npx ton-sovereign-deploy ./dist/ --domain mydapp.ton  # --watch is the default since v0.6

⚠ Honesty about --provider
v0.4 introduced --provider for "24/7 hosting via storage providers". It is fully implemented and signed end-to-end on mainnet (Round 1–7 post-mortem) — but the mainnet provider economy is currently dormant: the cheapest 5 providers have never sent accept_storage_contract in their entire on-chain history. Even foundation.ton self-hosts. So our default mode is --watch; --provider stays opt-in for the day mainnet providers come alive.
If you do try --provider and the contract fails to activate, run scripts/close-storage-contract.cjs <contract-addr> — the op::close_contract recovery path returns the locked balance.

Q1. Why does this exist?

DeFi UIs are routinely censored at the frontend layer. The contract survives, the user’s entry point doesn’t.

  • Tornado Cash — frontend removed by OFAC
  • Uniswap — token-level UI geofenced
  • 1inch · Balancer — domain takedowns
  • Root cause: regular DNS + regular hosting
⚡

Q2. What does it do?

One command, three layers of TON’s native infra wired together — no manual setup.

  • Bag upload via auto-installed storage-daemon
  • .ton DNS via TON Connect (QR + deeplink)
  • 24/7 hosting via storage provider contract
  • Vite · Next.js export · CRA build dirs auto-detected
🎯

Q3. Who is it for?

Builders whose frontend takedown would matter — not just “web3 curious.”

  • DeFi protocol teams hardening UI delivery
  • TON-native dApps wanting .ton as canonical
  • Censorship-exposed apps — journalism, privacy, DAO UIs
  • First public target: Gateway 2026 (May)

v0.5 lane B · landed 2026-05-10
--max-span uint8 cap is gone — provider contracts now take any uint32 span.
Instead of waiting on an upstream daemon fix, we found that the on-chain provider contract already accepts expected_max_span: uint32 — only the daemon CLI parser was buggy. We now build the offer-storage-contract BOC ourselves in TypeScript, using the daemon only to extract TorrentInfo and microchunk_hash. New CLI flag: --span <seconds> (default 86 400 = 1 day, max 4 294 967 295 ≈ 136 years). Verified by a byte-equal parity test against the daemon CLI’s output, plus 11 TL-B unit tests — 77/77 green.
no upstream dependency parity proven mainnet soak still pending

Known issues & tracking v0.5 inbox

✓ daemon --max-span uint8 bug — routed around
On-chain contract takes uint32; only the CLI parser was buggy. v0.5 builds the BOC in TS with @ton/core, taking any uint32 span (default 86400 = 1 day). Verified by byte-equal parity test against the daemon CLI’s 200s output.
resolved: 2026-05-10 · CLI flag --span · docs/v0.5/lane-b-self-generated-boc.md
P2P timeout on --provider <addr>
ADNL handshake to the provider often fails on mainnet. Workaround uses manual --rate + --max-span 200, sidestepping the P2P path.
blocks: per-provider rate negotiation
Public gateway propagation UX
ton.run returns 404 until the bag propagates. We show a warning, but no progress signal — needs a poll loop with peer-count + first-seen indicators.
target: v0.5 · pairs with --watch
Multi-provider contract
Single-provider hosting is a SPOF. Sign N contracts in parallel for redundancy, surface aggregate health.
target: v0.5 / v0.6
Testnet provider gap
Testnet has 171 registered providers but zero live ADNL nodes. Already gated with an early-exit warning when --testnet + --provider are combined.
status: guarded
Windows Defender warning on first run
Initial storage-daemon.exe download triggers AV warnings. Documented in README; consider Authenticode signing later.
target: post v1.0
